## Project: Discord Integration Reliability
- **Audit Findings**: Repo-Man identified that OpenClaw currently treats Discord REST API 2xx acceptance as success without verifying if components (polls/buttons) actually rendered on the platform.
- **Reliability Plan**: Scribe generated a 4-phase plan:
    - **WS1**: Delivery Verification (Message ID capture + post-send re-fetch).
    - **WS2**: Retry & Resilience (Exp backoff, rate-limit queuing).
    - **WS3**: Observability (Canary checks, structured logging).
    - **WS4**: Test Coverage (Integration/smoke tests).
- **Status**: Audit (WS1) initiated via `/audit` in `#discord-integration-reliability`.

## Tooling Discovery: Discord Polls
- **Issue**: Native `message` tool for polls consistently failed with generic errors. 
- **Workaround**: Discovered that the `openclaw message poll` CLI via `exec` is highly reliable when the internal tool fails.
- **Constraint**: Discord native polls have strict character limits for questions and options. Lengthy text causes silent or generic API failures.
- **Resolution**: Shortened poll text and used CLI fallback to successfully post the fix-path poll in #ops-github.
- **Documentation**: Updated `TOOLS.md` and `memory/robert-prefs.md` with these constraints and the CLI workaround.
